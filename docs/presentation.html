00 The problem

Almost everything your app does is a call into another process.

…and that boundary is a black box. You see your side and logcat — not the transaction crossing the kernel.

// all of these are Binder IPC, app → system_server
startActivity(intent)   // IActivityManager
getSystemService(POWER_SERVICE)  // IPowerManager
contentResolver.query(uri)  // IContentProvider
bindService(...)  ·  yourAidl.doThing(x)  // your own AIDL
// and when one breaks, all the kernel hands you is dmesg — pids + errno, no method:
binder: 1234:1234 transaction failed 29189/-22, size 512-0
binder: 1234:1301 oneway spam detection, sender 1234:1301
binder_alloc: 1998: no async space left, dropping transaction
binder: 5678 thread pool exhausted (4/4), starving

You only ever see the kernel's version — a pair of pids and an errno. Never which call it was, who called what, or did it fail. That question is unanswerable from the app side.

01 Binder in a nutshell — one parcel, moved once
  • 1Marshal. Client packs arguments into a Parcel (a flat byte buffer).
  • 2Hand off. It goes to the kernel's binder driver — the one middleman for all IPC.
  • 3Copy. Kernel copies the parcel into the other process and wakes it up.
  • 4Unmarshal. Target reads the parcel back, dispatches, replies.

What actually travels

  • ◆Interface descriptor — a string, e.g. android.app.IActivityManager
  • ◆Transaction code — an int. 7, not startActivity
  • ◆The parcel — the argument bytes
  • ◆sync / oneway flag, sizes, sender pid
// you write   am.startActivity(intent)
// the kernel sees   descriptor "android.app.IActivityManager"  ·  code 7  ·  sync
02 Existing solutions

The tools stop one field short.

What you have today

  • ▸Perfetto / atrace / ftrace binder tracepoints — src, target, size, sync flag.
  • ▸dumpsys / logcat — app-level events, above the IPC layer.
  • ▸Open-source binder decoders — resolve method names, but the AIDL must be compiled in ahead of time.
  • ▸Reading the .aidl by hand — counting methods to guess the code.

Where each stops

  • ✕Tracepoints cannot read the parcel — so no interface descriptor, no method.
  • ✕You learn a transaction happened, never which call.
  • ✕No cheap, always-on, whole-device view of who calls whom.
  • ✕Failures (DEAD/FAILED_REPLY) surface as an errno with no name.
  • ✕Decoders that bake the AIDL in at build time can't re-decode old logs against a newer catalog — no flexibility.

The missing capability is parcel access. The descriptor and the args live inside the parcel — and standard tracing can't touch it.

02 The gap → the idea

Read the parcel in-kernel. Name the method later.

That split — fast dumb capture vs rich offline decode — is the entire architecture.

05 Driving it — the control app

Drive the root daemon from an app.

  • ◆Talks to the daemon over TCP 127.0.0.1:3491. The runtime runs as root — an app can't grant itself BPF — so it drives, never spawns.
  • ◆Start / Stop toggles capture live by flipping the in-kernel filter flags — no reattach, no restart.
  • ◆Pick the sink — console · logcat · both · none — plus DLT streaming and error capture, all live.
  • ◆Three tabs. Control · Filter (interface discovery) · Deploy (su, adb fallback).
